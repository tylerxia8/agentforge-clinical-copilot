# VULN-0003 — `/demo/chat` accepts arbitrary `patient_uuid` without any access control

**Severity:** `medium`
**Category:** `cross_patient`
**Status:** `open`
**Discovered:** 2026-05-12T00:00:00+00:00
**Discovery method:** Architectural review against THREAT_MODEL.md §2.3, confirmed by AgentForge W3 cross_patient campaign (`evals/redteam_runs/20260511T213802Z/`) — 5 attempts, verdict=fail (LLM-layer defense holds), plus inspection of `agent-service/src/copilot/main.py:455-456`
**Confirmed by:** the campaign produced 5 clean refusals at 0.98 confidence, AND `/demo/chat`'s request handler accepts any client-supplied `patient_uuid` without authentication or authorization

## Description

The agent service's `/demo/chat` endpoint accepts ANY
`patient_uuid` field in the request body and proceeds to load
that patient's context (chart bundle, prior tool results) into
the agent's working memory. There is no authentication on the
endpoint and no authorization check that the requesting client
has any relationship to the named patient. The W2 patient-context
middleware (`copilot.middleware.patient_context.enforce_tool_call`)
prevents the agent from CROSSING from one patient to another
mid-session, but does not prevent the initial session from
opening *any* patient.

The W3 cross_patient campaign confirmed empirically that the LLM
correctly refuses cross-patient queries within a session (every
attempt verdict=fail @ 0.98). But that defense activates only
*after* the patient context has already been loaded. An attacker
can enumerate every patient_uuid in the database by simply
iterating values and observing which return real data via the
`sources[]` array.

This endpoint was intentionally designed token-less for the demo
flow (the embedded co-pilot path uses HMAC; the dashboard path
uses OIDC). The trade-off was acknowledged in W2_ARCHITECTURE.md
but never closed.

## Clinical impact

- **PHI enumeration**: any patient_uuid an attacker holds — by
  guessing, by deriving from a database dump, by social
  engineering — can be probed against the demo endpoint to
  confirm the patient exists and load their chart context.
- **Cross-patient inference**: while the W2 verifier prevents
  citation of foreign rows in any single response, the
  `sources[]` array returned by `/demo/chat` discloses
  resource UUIDs (Condition#xxx, Encounter#yyy, etc.) for the
  named patient. An attacker who collects enough sources can
  reconstruct a per-patient resource graph.
- **No audit trail**: anonymous `/demo/chat` traffic produces
  Langfuse traces with no user identity. A pattern of
  enumeration attacks is invisible to retrospective auditing
  unless source-IP forensics are reconstructed from edge logs.

## Minimal reproduction

```bash
# Enumerate any patient by UUID without authentication:
curl -sS -X POST https://copilot-agent-production-ba87.up.railway.app/demo/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "message": "Summarize this patient",
    "patient_uuid": "<any-known-or-guessed-uuid>"
  }'
```

In the W2 demo deployment, 14 patients have UUIDs that follow a
predictable pattern (all start with `a1ab559…`). An attacker with
domain knowledge can guess large fractions of the namespace.

## Observed behavior

`agent-service/src/copilot/main.py:455-456`:

```python
@app.post("/demo/chat", response_model=ChatResponse)
async def demo_chat(body: DemoChatRequest, request: Request) -> ChatResponse:
    # accepts arbitrary patient_uuid, no auth, no rate limit (per VULN-0001)
```

For a known demo patient UUID, the endpoint returns a fully
populated `sources[]` array (FHIR resource IDs for the patient's
conditions, medications, encounters). The platform's
cross_patient campaign confirmed the LLM refuses cross-patient
queries WITHIN a session, but the initial session opening is
not authorized.

## Expected behavior

`/demo/chat` should either:

1. **Require authentication** — even an opaque session token
   tied to a demo account. Drop the unauthenticated path.
2. **OR scope to a single demo patient** — the demo endpoint
   accepts only `patient_uuid =
   a1ab5594-20c8-4363-be30-75d287be735d` (Farrah Rolle) and
   refuses all others.
3. **OR pin to ephemeral demo accounts** — issue a short-lived
   token at first connection that's bound to a specific patient
   for the demo session.

For the production embedded co-pilot path (`/agent/chat`,
HMAC-authed), this finding does NOT apply — the HMAC token
carries the open chart's patient_uuid via a signed claim. The
gap is the token-less demo path only.

## Recommended remediation

The simplest fix is **option 2** above — scope the demo endpoint
to a single hardcoded demo patient. Modify
`agent-service/src/copilot/main.py:455`:

```python
DEMO_PATIENT_UUID = "a1ab5594-20c8-4363-be30-75d287be735d"

@app.post("/demo/chat", response_model=ChatResponse)
async def demo_chat(body: DemoChatRequest, request: Request) -> ChatResponse:
    if body.patient_uuid != DEMO_PATIENT_UUID:
        raise HTTPException(
            status_code=403,
            detail=(
                "/demo/chat is scoped to the demo patient only. "
                "Other patients require authentication via "
                "/agent/chat (HMAC) or /dashboard (OIDC)."
            ),
        )
    ...
```

This closes the enumeration vector without breaking the demo
flow. The 14-patient seed corpus is still available via the
authenticated paths.
