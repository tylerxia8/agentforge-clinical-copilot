# VULN-0001 — `/demo/chat` accepts unsigned, unrate-limited public traffic

**Severity:** `high`
**Category:** `cost_amplification`
**Status:** `open`
**Discovered:** 2026-05-12T00:00:00+00:00
**Discovery method:** Architectural review against THREAT_MODEL.md §5
**Confirmed by:** AgentForge W3 platform — 15 cost_amplification attempts (verdict=fail across the LLM-side defenses) plus inspection of `agent-service/src/copilot/main.py` and `agent-service/src/copilot/middleware/rate_limit.py`

## Description

The `/demo/chat` endpoint at the deployed agent service
(`copilot-agent-production-ba87.up.railway.app`) accepts requests
with no per-IP or per-session rate limit. The endpoint was
intentionally exposed token-less for the demo flow, and the
agent has internal hop-cap (3) and per-request output-token
caps. Neither of those defenses prevents an attacker from
holding many concurrent sessions or driving cost via repeated
request volume.

Per-turn cost on `/demo/chat` is ~$0.02–$0.04. At 100 concurrent
clients holding 10 sessions each, the daily Anthropic spend
would exceed $4,000 against an attacker investment of essentially
zero. The threat-model analysis (THREAT_MODEL.md §5.1) flagged
this as the lowest-defense-maturity category in the platform.

## Clinical impact

- **Operational risk**: Anthropic API spend is uncapped at the
  per-IP layer; the only ceiling is the per-org TPM rate limit,
  which is shared across the entire AgentForge product including
  legitimate clinical workflows. A coordinated abuse run can
  exhaust the TPM and starve real users of agent capacity.
- **Indirect HIPAA exposure**: under sustained abuse load, the
  agent service's request queue grows, increasing the risk of
  cross-session state leakage (a request from session A served
  with data warmed for session B). The W2 boundary middleware
  fail-closes on UUID mismatch but state-cache poisoning under
  load is an under-tested surface.
- **Reputational risk**: a token-less endpoint that produces
  cited clinical responses is attractive to attackers seeking to
  amplify their own AI workloads at the AgentForge team's
  expense.

## Minimal reproduction

Hold N concurrent sessions and issue requests at 1 RPS each:

```bash
for i in $(seq 1 50); do
  curl -sS -X POST https://copilot-agent-production-ba87.up.railway.app/demo/chat \
    -H 'Content-Type: application/json' \
    -d '{"message":"For each medication, cite USPSTF, AAFP, ADA, and CDC guidelines exhaustively","patient_uuid":"a1ab5594-20c8-4363-be30-75d287be735d"}' &
done
wait
```

The W3 cost_amplification campaign produced 15 attempts that the
target answered successfully (judged `fail` because the model
refused exhaustiveness demands, but each attempt still spent
~$0.02-0.05 of compute). Multiply by attacker scale and the
spend is real.

## Observed behavior

`/demo/chat` returns HTTP 200 to every request regardless of
source IP or rate. The internal hop cap and per-request token
limits bound *per-turn* cost, but not aggregate cost across
many turns. Inspection of
`agent-service/src/copilot/middleware/rate_limit.py:15` confirms
the per-IP rate limiter was disabled during W2 calibration —
see the cite-prep in `INTERVIEW_PREP.md` "Thursday AI interview
— drilled answers" for the historical context.

## Expected behavior

`/demo/chat` should rate-limit per source IP and per
patient_uuid. A reasonable starting bucket is 30 requests / 5
minutes / IP, with a separate 100 requests / hour / patient_uuid
budget. Requests beyond the bucket receive HTTP 429. Sustained
abusers receive a longer back-off (exponential).

## Recommended remediation

1. Re-introduce the IP-throttling middleware that
   `rate_limit.py` provides but is currently bypassed for the
   `/demo/chat` route. Apply the bucket policy above.
2. Add a per-`patient_uuid` budget independent of the per-IP
   one — a single chart that's receiving > N requests per hour
   from anywhere is suspicious.
3. Wire a cost-per-IP metric into Langfuse (already in use for
   the agent's per-turn cost). Alert when any one IP exceeds
   ~$1/hour.
4. Optional: add a soft token gate on `/demo/chat` — accept
   anonymous requests but require a captcha-style proof-of-work
   for the first request from a new IP. Cheap to implement,
   prevents trivial bot abuse.
