# `agent-service/vulns/` — deploy mirror

> ⚠️ **This directory is a deploy-time mirror.** The CANONICAL
> vulnerability reports live at the repo root `/vulns/` per the
> W3 PRD's required layout. This directory exists ONLY because
> the agent-service Dockerfile's build context is `agent-service/`,
> and the `/adversarial` visibility page needs the reports
> accessible from inside the Docker image.

## Sync workflow

Before each `railway up --service copilot-agent`, refresh the
mirror from canonical:

```bash
# From repo root:
rm -rf agent-service/vulns
cp -r vulns agent-service/vulns
```

That single command keeps the deployed `/adversarial` page in
sync with the latest vuln reports.

## Why we don't symlink

Symlinks would be the ideal solution (single source of truth,
no drift). But:
- Windows requires admin or developer mode to create symlinks
- `git` checkout behavior on symlinks is platform-dependent
- Some CI runners materialize symlinks as the linked file
  (loses the symlink semantics)

Mirror + manual sync is more portable.

## Why we don't change Railway's rootDirectory

A cleaner long-term fix: flip the `copilot-agent` service's
Railway rootDirectory from `agent-service` to `.` (repo root)
and update the Dockerfile's COPYs to be `agent-service/`-
prefixed. Then `vulns/` would be in the build context naturally,
no mirror needed.

This is a Railway service-config change that requires the
Railway dashboard (not done from CLI yet). Tracked as follow-on
work — see commit history near 107c045 for the build-context
discussion.

## Drift risk

Real but manageable. Vuln reports are write-once (you don't
usually edit a report after filing). The Documentation Agent
writes new reports to the CANONICAL repo-root `/vulns/`, never
to this mirror. The mirror only updates when the operator
explicitly syncs.

A future check could be added at `pytest agent-service/tests/`
time that diffs the two dirs and fails on drift. Not blocking
for the W3 final submission.
